## в—Ҳ Why I Built This

Every interviewing tool I've touched falls into one of two traps вҖ” a glorified textarea with no context, or a bloated enterprise SaaS I'd never actually own.

I wanted one browser tab where a real technical interview can happen end-to-end. A VSCode-grade editor. A live video call. Code that actually runs. A real problem to solve. Chat in the same screen. And the ability to see the session history afterwards.

Code\_Tester is that tab. It's built by me, runs on my stack, and is exactly as complex as it needs to be вҖ” no more.

- **Host** creates a session, picks a problem & difficulty (easy / medium / hard)
- A unique `callId` is generated; a **Stream Video call** and **Stream Chat channel** are created simultaneously
- Any authenticated user with the link can **join** вҖ” they're auto-added to the session and chat channel
- Code runs against **Judge0 CE** (proxied through Express) вҖ” C++ 13.2, Java 17, Python 3.11, Node 18
- The host can **End Session** вҖ” this hard-deletes both the video call and the chat channel, and marks the session `completed` in MongoDB

### Solo Practice

Every problem in the library is also available for solo practice at `/problems/:contestId/:index`. The layout is a resizable two-pane split: problem statement on the left, Monaco + console on the right. Two buttons: **Run Code** (runs against sample input) and **Submit** (launches confetti on pass).

### Problem Library

`/problems` is a full browsable problem set:

- Live-fetched from the **Codeforces API** and cached server-side for 15 minutes
- Falls back to a curated local list of ~80 problems if the API is unreachable
- Filterable by full-text search, tag, and difficulty tier (Easy вүӨ 1200 В· Medium 1300вҖ“1900 В· Hard вүҘ 2000)
- Paginated at 50 problems per page
- Clicking any problem opens the solo practice workspace

### Dashboard

After login, the dashboard shows:

- **Live Rooms** вҖ” active sessions polled every 5 seconds, filtered to within the last 2 hours
- **Past Sessions** вҖ” completed sessions where you were host or participant
- **Stats** вҖ” active rooms, completed sessions, unique problems attempted, success rate
- Animated count-up numbers triggered by scroll via `IntersectionObserver`

## в—Ҳ API Routes

| Method | Route | Auth | What it does |
|---|---|---|---|
| `POST` | `/api/user/save` | рҹ”’ Firebase | Upsert user in MongoDB + sync to Stream Chat |
| `GET` | `/api/user/profile` | рҹ”’ Firebase | Return combined Firebase + DB profile |

| `GET` | `/api/session/active` | рҹ”’ Firebase | Last 20 active sessions |
| `GET` | `/api/session/recent` | рҹ”’ Firebase | Completed sessions where user is host or participant |

## в—Ҳ Background Jobs вҖ” Inngest

Two event-driven functions handle user lifecycle automatically, so the DB and Stream Chat always stay in sync with Firebase Auth:

```
Event: user.created
  в””в”Җ sync_user_data
       в”ңв”Җ step: connect-db
       в”ңв”Җ step: create-user-db     вҶ’ User.create({ firebaseUID, name, email, profileImage })
       в””в”Җ step: create-stream-user вҶ’ chatClient.upsertUser(...)

Event: user.deleted
  в””в”Җ delete_user
       в”ңв”Җ step: connect-db
       в”ңв”Җ step: delete-db-user     вҶ’ User.findOneAndDelete({ firebaseUID })
       в””в”Җ step: delete-stream-user вҶ’ chatClient.deleteUser(userId)
```

To test locally, run the [Inngest Dev Server](https://www.inngest.com/docs/getting-started/local-development):

```bash
npx inngest-cli@latest dev
```

## в—Ҳ Code Execution вҖ” Judge0

Code submitted in any session or problem goes through a backend proxy at `POST /api/problems/execute`. The backend calls Judge0 CE directly вҖ” the frontend never touches Judge0.

| Language | Judge0 ID | Runtime |
|---|---|---|
| C++ | `105` | GCC 13.2.0 |
| Java | `91` | OpenJDK 17.0.1 |
| Python | `92` | Python 3.11.2 |
| JavaScript | `93` | Node.js 18.15.0 |

The proxy maps Judge0's response shape into a standard `{ run: { stdout, stderr, code, output } }` object that both `OutputPanel` and `ProblemDetail` consume.

## в—Ҳ Auth Flow

```
User signs in (Firebase client SDK)
   в”Ӯ
   в”ңв”Җ Email/Password  вҶ’  signInWithEmailAndPassword
   в””в”Җ Google          вҶ’  signInWithPopup(GoogleAuthProvider)
           в”Ӯ
           в–ј
    user.getIdToken()  вҶ’  Bearer token
           в”Ӯ
           в–ј
    POST /api/user/save   (Authorization: Bearer <token>)
           в”Ӯ
           в–ј
    backend: verifyFirebaseToken middleware
         вҶ’ admin.auth().verifyIdToken(token)
         вҶ’ attaches req.user  (Firebase claims)
         вҶ’ attaches req.user._id  (MongoDB ObjectId, if found)
           в”Ӯ
           в–ј
    User upserted in MongoDB + Stream Chat
```

All protected routes use the same `verifyFirebaseToken` middleware. The frontend sends the Firebase ID token on every request via `Authorization: Bearer`.
